# v3 UX — Design Spec (build contract)

**Date:** 2026-06-08 · **Branch:** `v3-ux` · **Status:** approved, building

A new web UX shipped **in parallel** with the existing `ui`/`darwin`/`os9`/`osish`
bundles. Left-hand collapsible sidebar of hook sources; each hook item expands to
its matched routines; each `(item × routine)` is one chat with its own queue, built
from **prompt-kit** components. Plus reused Deliveries / Routines / Settings / About
views. Nothing in the current UX changes.

This document is the shared contract. Every implementer reads it and imports from the
shared component inventory (§5) rather than reinventing.

---

## 1. Goals / Non-goals

**Goals**
- 5th build target `web/v3/` served at `/v3/`. Zero edits to existing bundles.
- Sidebar: 5 collapsible hook sections — **Schedules** (cron routines), **Errors**
  (Sentry), **Alerts** (Datadog), **Tickets** (Linear — *stub*, §4a), **Pull Requests**
  (GitHub).
- Each section lists items by key (PR#+title / Sentry issue id / Datadog monitor /
  routine name). Each item expands to its matched routine markdown files. Each
  `(item × routine)` = one chat thread + queue.
- Chat pane built from prompt-kit, rendering the agent's transcript as rich agentic
  blocks (reasoning, tool calls, sources), interactive composer from day one.
- Bottom-nav views: Deliveries, Routines, Settings, About.

**Non-goals**
- No changes to webhook ingestion, matching, or the queue worker's run semantics
  (only one additive branch for `web:message`, §8).
- No new persistence store; reuse `hook-queue.db`, `sessions.json`, jsonl transcripts.
- Feedback Bar is render-only (no-op persistence) in v1.

## 4. Sidebar tree (client-side, from existing queue API)

Source data: `GET /api/hooks/queue` (snapshot) + `GET /api/hooks/queue/events` (SSE
deltas). `QueuedMessage` fields used: `event`, `jobName`, `scope`, `threadId`,
`prRepo`, `prNumber`, `keys`, `fields`, `outcome`, `status`, `payload`.

`lib/tree.ts` builds:

```
SidebarTree = Section[]
Section { source: "github"|"sentry"|"datadog"|"routines", label, items: Item[] }
Item     { key: string, title: string, routines: ThreadRef[] }   // key = PR#/issue id/monitor/routine
ThreadRef{ threadId, jobName, status, outcome, lastAt }
```

Mapping:
- **source** from `event` prefix: `github` (pull_request*, issue*, …), `sentry:*`,
  `datadog:*`; everything non-hook (plain cron run) → `routines`.
- **item key/title**: GitHub → `prRepo#prNumber` + `keys.key2`; Sentry/Datadog →
  `keys.key1`; routines → `jobName`.
- **routines under an item**: group remaining rows by `jobName`; each → `ThreadRef`
  with `threadId = <jobName>:hook:<scope>`.
- Status badge per ThreadRef from `status`/`outcome` (queued/running/ok/pass/error)
  — reuse the badge logic already in `web/ui/sections/PrsSection.tsx`.

Collapse state persisted in `localStorage` (`clawdcode:v3:collapsed`).

### 4a. Linear tickets (stub)

A fifth source, **Tickets**, sourced from Linear. Trigger condition: a Linear issue or
comment that **@mentions the bot's Linear user**. Stub scope for this build:
- Extend `DeliverySource` in `shared/deliveryTypes.ts` to `… | "linear"`.
- Add a **stub** receiver `POST /api/webhooks/linear` in `src/hooks/` mirroring
  `sentry.ts`/`datadog.ts` shape, that: verifies the Linear webhook signature, parses the
  payload, and **only enqueues when the bot is @mentioned** (scan issue description /
  comment body for the bot's mention) — otherwise returns `ok` with no match. Mark
  clearly as a stub (`// STUB: Linear @mention trigger`); full field extraction
  (`keys`/`fields`) can be minimal (identifier + title).
- Tree mapping (§4): `event` prefix `linear:*` → `tickets` section; **item key** =
  ticket identifier (e.g. `ENG-204`); **title** = ticket title; routines grouped by
  `jobName` as usual. Section glyph `❏`, status dots identical to other sources.
- Register `linear` in the v3 sidebar section list + the existing `/api/webhooks/*`
  pre-auth routing block in `server.ts`.

**Vendoring approach:** clawdcode is Bun+esbuild + Tailwind v4 + DaisyUI, **not** a
shadcn/Next app — do **not** run `npx shadcn add` against the repo. Instead copy each
component's source from `https://prompt-kit.com/c/<name>.json` (the `files[].content`
field) into `components/prompt-kit/`, add the minimal shadcn primitives it imports
into `components/ui/`, and rewrite `@/` import paths to relative. Restyle to darwin-ui
by mapping Tailwind/shadcn color tokens to DaisyUI (`bg-base-100`, `text-base-content`,
`border-base-300`, `text-primary`, etc.). Deps to add to `package.json`: `shiki`,
`react-markdown`, `remark-gfm`, `remark-breaks`, `use-stick-to-bottom`.

## 6. Chat pane — transcript → parts

`GET /api/v3/threads/:threadId/messages?limit&offset` returns **structured** parts
(not flattened text). Backend resolves `threadId → sessionId` via `sessionManager`,
reads the jsonl transcript (`~/.claude/projects/<cwd>/<sessionId>.jsonl`), and emits:

```ts
// lib/transcriptParts.ts (shared)
type ChatPart =
  | { kind: "system";    id; text }                    // hook trigger / SystemMessage
  | { kind: "text";      id; role:"user"|"assistant"; markdown:string }
  | { kind: "reasoning"; id; markdown:string }         // thinking block
  | { kind: "tool";      id; tool: ToolPart }          // tool_use + matched tool_result
  | { kind: "sources";   id; sources: SourceLink[] }   // appended per assistant turn
type SourceLink = { href:string; label:string; title?:string }
```

Parser rules: assistant content blocks → `text`(text) / `reasoning`(thinking) /
`tool`(tool_use, paired with the following `tool_result` by `tool_use_id`); user turns
→ `text` role user (skip tool_result-only user turns, they fold into the tool part).
`sources` derived by `lib/sources.ts`: hook origin URL from the thread's first queue
row `payload` (PR `html_url` / Sentry issue url / Datadog monitor url) + `file_path`
inputs from tool calls → `file:line` labels.

`ChatPane` renders `ChatContainerRoot > ChatContainerContent` mapping parts to the
§5 components, `ScrollButton`, then the composer.

## 7. Live streaming

`GET /api/v3/threads/:threadId/stream` (SSE via existing `sseResponse` helper):
- Emits a `snapshot` of current parts, then `append`/`update` deltas as the active
  session jsonl grows (tail by byte offset; re-parse new lines into `ChatPart`s).
- Merges queue-status deltas for the thread (queued/running/done) so the pane can
  show `Loader`/`ThinkingBar` while `status=running` before tokens land.
- `useThreadStream.ts` owns the EventSource (token via `getApiToken()`), reconnect,
  and merge into pane state.

No change to the worker's spawn path — we observe the transcript it already writes.

## 8. Interactive composer

`POST /api/v3/threads/:threadId/message` body `{ text }`:
- `hookQueue.enqueue({ id:"user-"+uuid, threadId, jobName:<from thread>, event:"web:message",
  scope:<thread scope>, payload:{ type:"user-message", text }, prRepo, prNumber })`.
- The existing worker (`drainHookQueue → claimThread → runQueuedBatch → runJob{hookScope}
  → execClaude --resume`) picks it up. **One additive change** in
  `buildCoalescedHookPrompt` (src/commands/start.ts ~1509): when a message's
  `event === "web:message"`, render its block as the raw `payload.text` (a user turn),
  not a hook summary. jobName/scope resolved from any existing queue row for the thread.
- Optimistic echo in the pane; the SSE stream reconciles once the worker runs.

## 9. Bottom-nav views

- **Deliveries** (`DeliveriesView`): real-time incoming-hook log from
  `/api/hooks/deliveries` + `/api/hooks/events` (SSE). Row inspect via
  `/api/hooks/deliveries/:id/payload`. **Jump-to-chat**: resolve `matched[]` job names
  + delivery scope → `threadId`(s); clicking selects that thread in the navigator and
  switches the main pane to the chat. Reuse delivery rendering patterns from
  `web/ui/sections/DeliveriesSection.tsx`.
- **Routines** (`RoutinesView`): markdown editor + git sync. Reuse the routine editor
  component + `/api/jobs/file*` CRUD + `/api/jobs/repos/:slug/{pull,sync}` from the
  existing JobsSection wiring, in the v3 shell.
- **Settings** / **About**: import and render the existing
  `web/ui/sections/SettingsSection` / `AboutSection` (they depend only on `web/api/*`
  + darwin-ui, both available in v3). Thin wrappers only.

## 14. Design system — "Abyssal Operator Console" (reskin contract)

clawdcode 🦞 watches the deep for webhooks and dispatches agents. The visual identity is
a **bioluminescent deep-sea console**: live/running work glows **lobster-coral** and
breathes; resolved work settles to **bioluminescent teal**. Reference mockup:
`docs/superpowers/specs/2026-06-08-v3-ux-mockup.html`.

**Fidelity:** full reskin — port tokens, fonts, motion, and per-component styling so v3
matches the mockup. **Themes:** two, via `data-theme` on `<html>`, default `abyssal`,
toggle persisted in `localStorage('clawdcode:v3:theme')`:
- `abyssal` (dark, default) — deep-sea night.
- `tidepool` (light) — sunlit shallows over warm sand.

**Type** (Google Fonts; no Inter/Roboto/system): display **Instrument Serif** (section
headers, thread titles, the live shimmer label); UI **Hanken Grotesk**; mono
**JetBrains Mono** (keys, ids, timestamps, code, badges — the actual subject matter).

**Tokens** (CSS variables; same names both themes so component CSS is theme-agnostic):

| token | abyssal (dark) | tidepool (light) | use |
|---|---|---|---|
| `--abyss-0/1/2` | `#0c1417 / #101a1e / #13201f` | `#f4efe4 / #ede6d6 / #fbf7ee` | app/sidebar bg layers |
| `--surface` / `--surface-2` | `#172428 / #1c2d31` | `#fffcf5 / #f3ecdd` | panels / raised |
| `--line` / `--line-2` | `#26393e / #324a50` | `#ddd3be / #c9bba0` | hairlines |
| `--bone` / `--bone-dim` / `--bone-faint` | `#ece6d8 / #9fb0b0 / #5e7274` | `#14211f / #4a5a57 / #8a9591` | text ramp |
| `--coral` / `--coral-deep` | `#ff5a3c / #ff3b2e` | `#ef4a2c / #d63a1f` | primary signal: live/running |
| `--cyan` / `--cyan-dim` | `#3fe0bf / #2a9d89` | `#0e9e86 / #0b7d6a` | resolved/links/ok |
| `--amber` | `#f2b84b` | `#c8881f` | pending |
| `--red` | `#ff4d4d` | `#d63a3a` | error |

**Status language (universal):** coral = running (`@keyframes biolum` breathing glow);
teal = ok/resolved; amber = pending; red = error; faint = idle. Active thread gets a
coral left-spine + glow.

**Motion:** staggered page-load reveal (sidebar sections cascade, stream parts rise);
coral bioluminescent pulse on running dots/chips; serif shimmer on the live Thinking Bar;
tool/reasoning expand with eased max-height. Honor `prefers-reduced-motion`.

**Atmosphere:** layered radial depth gradient (coral + teal light wells), faint SVG
grain overlay, inset vignette. Tidepool keeps the same structure at low opacity (paper
grain, soft warm wells) so it reads as sunlit water, not a flat light theme.
